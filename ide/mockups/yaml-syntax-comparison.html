§1 Step Syntax

Each step follows a consistent structure: id → identify → label → configure → returns → validate. The uses: field uses hierarchical namespaces.

steps: - id: send_notification uses: connector/http/call_via_dataplane name: Send notification to SUT with: method: POST dataplane_url: ${{ vars.dataplane_url }} path: "/companycertificate/available" edr_token: ${{ vars.edr_token }} body: header: messageId: "urn:uuid:${{ vars.avail_message_id }}" senderBpn: ${{ env.consumer_bpn }} returns: status_code: type: number class: status_code response_body: type: object class: response_body response_headers: type: object class: response_headers validate: - uses: validate/assert with: input: status_code operator: equals value: 200 - uses: validate/field with: input: response_body path: "header.messageId" operator: matches_regex value: "^urn:uuid:[0-9a-f-]{36}$" - uses: validate/field with: input: response_body path: "content.requestStatus" operator: one_of value: - "IN_PROGRESS" - "COMPLETED" - "REJECTED"

§2 Syntax Reference

Each keyword has a clear purpose and origin. TestLab extends GitHub Actions vocabulary with domain-specific testing primitives.

KeywordMeaningOriginExample
id: Unique step identifier within the test. Auto-generated by IDE, user can override. Used for qualified variable references. GitHub Actions id: discover
uses: Identifies which step type to execute. Uses hierarchical namespaces: namespace/[sub-namespace/]action GitHub Actions uses: connector/consumer/negotiate
name: Human-readable label for the step GitHub Actions name: Send notification to SUT
with: Input parameters passed to the step GitHub Actions with: { method: POST, path: "/..." }
validate: Assertions to run after step execution. Array of assertion objects using the same uses: / with: pattern as steps. The validate/ namespace contains all assertion types. TestLab validate: [{ uses: validate/field, with: { input: body, path: "x", operator: equals, value: 1 } }]
${{ vars.x }} Test-local variable — auto-persisted from step returns:. Scoped to current test only. GitHub Actions ${{ vars.edr_token }}
${{ env.x }} Environment variable reference — resolves to external configuration (connector addresses, BPNs, timeouts) GitHub Actions ${{ env.provider_address }}
returns: Declares step return values — all automatically persisted as variables. Each entry declares type: (data type) and class: (semantic type). "Steps are functions" — functions return values. TestLab returns: { token: { type: string, class: auth_token } }
type: Data type of the output variable (string, number, boolean, object, array) GitHub Actions type: string
class: Semantic type annotation for a variable — enables IDE dropdown filtering and type-safe connections between steps TestLab + ADR-0009 class: uuid | url | bpn | auth_token
service: References a service defined in the TCK env.services by name GitHub Actions adapted service: provider_edc
env: Environment configuration block — contains variables (with typed classes), services, and schemas GitHub Actions env: { variables: {...}, services: [...], schemas: {...} }
preconditions: Reusable policies and contracts, referenced by ID in tests TestLab preconditions: [{ id: ccmapi_usage_policy, uses: policy }]
Functions: Steps with no with: inputs that auto-generate typed returns. For example, util/generate_uuid produces a variable with class: uuid automatically. Functions are pure generators — no configuration needed. All return values are automatically persisted as ${{ vars.x }}.

§2c Assertion Operators

The validate: field uses the same uses: / with: pattern as steps. Assertions live in the validate/ namespace — just like connector/, mock/, or util/ are step namespaces.

OperatorDescriptionValue TypeExample
equals Equals scalar operator: equals, value: 200
not_equals Not equals scalar operator: not_equals, value: "ERROR"
not_null Value exists and is not null (no value needed) operator: not_null
matches_regex Matches regular expression string (regex) operator: matches_regex, value: "^urn:uuid:[0-9a-f-]{36}$"
one_of Value is one of the listed options array operator: one_of, value: ["A", "B", "C"]
contains String/array contains value scalar operator: contains, value: "certificate"
gt Greater than number operator: gt, value: 0
gte Greater than or equal number operator: gte, value: 1
lt Less than number operator: lt, value: 500
lte Less than or equal number operator: lte, value: 100

Assertion Types (validate/ namespace)

Important: The input field always references a variable declared in a step's returns: block. At runtime it resolves to ${{ vars.step_name.field }}.
uses: valuePurposeRequired with: fields
validate/assert Compare a direct return variable against scalar/array input, operator, value
validate/field Compare a nested field within a return variable (dot-path navigation) input, path, operator, value (optional for not_null)
validate/object Deep-compare return variable against a JSON object input, operator, value (object)
validate/schema Validate return variable against a JSON schema input, schema
Harmonized syntax: Assertions use the exact same uses: + with: pattern as steps. The validate/ namespace is a namespace like connector/ or mock/. One mental model for the entire YAML file.

§4 Migration from v1 to v2

Complete guide for migrating existing TestLab YAML files to the new GitHub Actions-inspired syntax.

Quick Reference

v1 (current) v2 (new) type: edc_call_dataplane → uses: connector/http/call_via_dataplane type: edc_negotiate → uses: connector/consumer/negotiate type: http_call → uses: http/call type: generate_uuid → uses: util/generate_uuid type: mock_create_endpoint → uses: mock/create_endpoint description: "Send notification" → name: Send notification params: → with: @step_output → ${{ vars.step_output }} @config_value → ${{ env.config_value }} store_in_memory: { key: "value" } → returns: { key: { type: string, class: type } } services: (top-level) → env: { services: [...] } validate: ["NOT_NULL:output"] → validate: [{ uses: validate/assert, with: { input: x, operator: not_null } }]

Migration Rules

  1. Replace type: with uses: on every step — use namespaced identifiers (e.g., connector/provider/create_asset)
  2. Replace description: with name:
  3. Replace params: with with:
  4. Replace @variable_name with ${{ vars.variable_name }} when it comes from a previous step's returns
  5. Replace @variable_name with ${{ env.variable_name }} when it comes from external configuration (connector addresses, BPNs, timeouts)
  6. Convert validate: from string arrays to assertion objects using uses: validate/... + with: { input, operator, value }. The input field always references a variable from the step's returns: block.
  7. In services: replace type: with uses: and config: with with:
  8. Move services: from top-level into env: { services: [...] }
  9. Rename keyword: store_in_memory: → returns: and add type annotations: returns: { key: { type: string, class: <semantic_type> } }
  10. Add class: to each env.variables entry to declare its semantic type

Steps — Before & After

v1 (current)
steps: - type: pull_data_filtered_from_precondition description: Discover notification endpoint params: service: provider_edc counter_party_id: "@provider_bpn" counter_party_address: "@provider_address" validate: - "NOT_NULL:edr_token" store_in_memory: edr_token: "edr_token" dataplane_url: "dataplane_url"
v2 (new)
steps: - uses: connector/consumer/pull_data_filtered name: Discover notification endpoint with: service: provider_edc counter_party_id: ${{ env.provider_bpn }} counter_party_address: ${{ env.provider_address }} validate: - uses: validate/assert with: input: edr_token operator: not_null returns: edr_token: type: string class: auth_token dataplane_url: type: string class: dataplane_url

Services — Before & After

v1 (current)
services: - name: provider_edc type: edc_connector_saturn config: base_url: "@provider_address" dma_path: /management
v2 (new)
env: services: - name: provider_edc uses: edc_connector_saturn with: base_url: ${{ env.provider_address }} dma_path: /management

§5 TCK Definition File

The TCK file defines the test suite context: environment variables (with typed classes), shared services, and preconditions. Individual test files inherit all env: variables — no cherry-picking.

kind: tck name: certificate-management version: "1.0" description: > Validate CCMAPI certificate management per CX-0135 v3.1.0 author: Certificate Management Expert Group standards: - id: CX-0135 version: v3.1.0 tags: - CCM env: # Variables — available as ${{ env.variable_name }} variables: provider_address: class: url description: Provider EDC DSP endpoint required: true provider_bpn: class: bpn description: Provider BPNL required: true consumer_bpn: class: bpn description: Consumer BPNL (TestLab company) required: true certificate_type: class: string description: Certificate type to request default: "iso9001" sut_response_timeout: class: number description: Seconds to wait for SUT response default: 300 # Services — available by name in steps (service: provider_edc) services: - name: provider_edc uses: edc_connector_saturn with: base_url: ${{ env.provider_address }} dma_path: /management - name: testlab_edc uses: edc_connector_saturn with: base_url: ${{ env.testlab_management_url }} dma_path: /management # Schemas — imported from schemas/ folder schemas: certificate_request: file: schemas/certificate_request.yaml certificate_response: file: schemas/certificate_response.yaml # Preconditions — reusable policies/contracts preconditions: - id: ccmapi_usage_policy uses: policy name: CX-0135 ODRL usage policy with: dataspace_version: saturn policy_type: usage permissions: - action: use constraints: - left_operand: UsagePurpose operator: eq right_operand: "cx.ccm.base:1" # Test suite tests: - test: tests/request_certificate.yaml name: Request certificate with full header envelope - test: tests/available_notification.yaml name: Send available notification

§5b Variable Scoping & Persistence Model

Variables live in two scopes. All returns: are automatically persisted — no flags needed.

Scope Namespace Lifetime How Populated How Consumed
Test-local ${{ vars.x }} Current test only Every returns: value from any step Downstream steps in the same test via ${{ vars.x }}
TCK-global ${{ env.x }} Entire TCK run (all tests) Declared in TCK env.variables: or exported from a test All tests via ${{ env.x }} — imported automatically
Key rules:
  • returns: values are always persisted in test memory — no opt-in flag.
  • To share a return value across tests → export it to env (promotes from test scope to TCK scope).
  • When a test imports a TCK, it receives all env: variables — no partial imports.
  • vars are ephemeral per test. env persists across the TCK run.

§5c IDE Variable Blocks — Display Rules

The IDE uses two distinct block types for variable references. Blocks do NOT display scope prefixes — the scope is implicit from the block type.

Block TypeDisplayEditable?SourceSerializes As
env_ref Dropdown showing plain name (e.g., consumer_bpn) User selects from dropdown TCK manifest env.variables ${{ env.consumer_bpn }}
vars_ref Read-only label showing plain name (e.g., status_code) NOT editable — auto-generated from step returns Preceding steps' returns ${{ vars.status_code }}
Key UX rules:
  • NO env. prefix displayed on env_ref blocks — user sees just consumer_bpn
  • NO vars. prefix displayed on vars_ref blocks — user sees just status_code
  • The scope (${{ env.x }} vs ${{ vars.x }}) is determined by the block type, not shown in the UI
  • vars_ref blocks are created automatically when a step declares returns — never manually defined
  • Block colors distinguish scope: teal for env, peach for vars

Visual Representation

§6 Variable Reference — Finalized Decision

The finalized variable syntax uses Option B: Flat ${{ vars }}. This provides scope visibility with GHA-familiar syntax while keeping references concise.

# Finalized variable reference syntax (ADR-0010) steps: - id: discover uses: connector/consumer/pull_data_filtered with: counter_party_id: ${{ env.provider_bpn }} returns: edr_token: type: string class: auth_token - uses: connector/http/call_via_dataplane with: edr_token: ${{ vars.edr_token }} # flat ref (unambiguous) sender: ${{ env.consumer_bpn }} # env ref

Resolution Rules

PatternScopeResolution
${{ vars.x }} Test-local Search preceding steps' returns for key x. Single match → resolved. Zero/multiple → compiler error.
${{ vars.step_id.returns.x }} Test-local (qualified) Direct reference to a specific step's return value. Used when flat ref is ambiguous.
${{ env.x }} TCK-global Resolves from TCK manifest env.variables. Available to all tests.
${{ env.schemas.name }} TCK-global References an imported schema object from env.schemas.
Why Option B was chosen:
  • Explicit scope visibility (vars vs env) catches errors at author-time
  • GHA-familiar syntax reduces learning curve
  • Qualified form (vars.step_id.returns.x) handles ambiguity without requiring id on every step
  • Compile-time detection of undefined references

§8c Schema Imports — schemas/ Folder

Schemas define the expected structure of request/response payloads. They live in the schemas/ folder next to tests and are imported at the TCK level.

Folder Structure

certificate-management-v1.0/ ├── index.yaml # TCK manifest (kind: tck) ├── schemas/ │ ├── certificate_request.yaml # Request body schema │ ├── certificate_response.yaml # Response body schema │ └── notification_header.yaml # Notification envelope └── tests/ ├── request_certificate.yaml └── available_notification.yaml

Schema File Example

# schemas/certificate_request.yaml kind: schema name: certificate_request version: "1.0" description: CX-0135 certificate request payload structure type: object required: - header - content properties: header: type: object required: [messageId, senderBpn, recipientBpn] properties: messageId: type: string format: urn:uuid senderBpn: type: string pattern: "^BPN[LS][0-9A-Z]{12}$" recipientBpn: type: string pattern: "^BPN[LS][0-9A-Z]{12}$" content: type: object required: [certificateType] properties: certificateType: type: string enum: [iso9001, iatf16949, iso14001]

Importing Schemas in TCK Manifest

env: variables: # ... (as before) # Schema imports — loaded from schemas/ folder schemas: certificate_request: file: schemas/certificate_request.yaml certificate_response: file: schemas/certificate_response.yaml notification_header: file: schemas/notification_header.yaml

Using Schemas in Steps

steps: - id: wait_request uses: mock/create_endpoint name: "Wait for certificate request" with: method: POST path: /api/v1/certificates request_schema: ${{ env.schemas.certificate_request }} returns: response_body: type: object class: response_body validate: - uses: validate/schema with: input: response_body schema: ${{ env.schemas.certificate_response }}
Schema usage patterns:
  • request_schema in with: — validates incoming requests against the schema (mock endpoints)
  • schema in validate: — asserts a return value matches the schema structure
  • Referenced as ${{ env.schemas.name }} — schemas are TCK-global, available to all tests
  • Schema files use kind: schema — the compiler resolves and loads them at compile time
