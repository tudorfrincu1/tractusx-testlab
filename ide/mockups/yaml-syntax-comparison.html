§1 Step Syntax

Each step follows a consistent structure: identify → label → configure → validate → capture.

steps: - uses: http_call_dataplane name: Send notification to SUT with: method: POST dataplane_url: ${{ vars.dataplane_url }} path: "/companycertificate/available" edr_token: ${{ vars.edr_token }} body: header: messageId: "urn:uuid:${{ vars.avail_message_id }}" senderBpn: ${{ env.consumer_bpn }} validate: - type: EQUALS output: status_code value: 200 - type: NOT_NULL output: response_body

§2 Syntax Reference

Each keyword has a clear purpose and origin. TestLab extends GitHub Actions vocabulary with domain-specific testing primitives.

KeywordMeaningOriginExample
uses: Identifies which step type to execute (like an action reference) GitHub Actions uses: mock_endpoint
name: Human-readable label for the step GitHub Actions name: Send notification to SUT
with: Input parameters passed to the step GitHub Actions with: { method: POST, path: "/..." }
validate: Assertions to run after step execution TestLab validate: [{ type: EQUALS, output: status_code, value: 200 }]
${{ vars.x }} Local variable reference — resolves to values produced by previous steps (outputs) GitHub Actions ${{ vars.edr_token }}
${{ env.x }} Environment variable reference — resolves to external configuration (connector addresses, BPNs, timeouts) GitHub Actions ${{ env.provider_address }}
outputs: Declares step outputs. Each entry declares type: (data type) and class: (semantic type). Replaces the old store_in_memory keyword. TestLab outputs: { token: { type: string, class: auth_token } }
type: Data type of the output variable (string, number, boolean, object, array) GitHub Actions type: string
class: Semantic type annotation for a variable — enables IDE dropdown filtering and type-safe connections between steps TestLab + ADR-0009 class: uuid | url | bpn | auth_token
service: References a service defined in the TCK env.services by name GitHub Actions adapted service: provider_edc
env: Environment configuration block — contains variables (with typed classes) and services available to all tests GitHub Actions env: { variables: {...}, services: [...] }
preconditions: Reusable policies and contracts, referenced by ID in tests TestLab preconditions: [{ id: ccmapi_usage_policy, uses: policy }]
Functions: Steps with no with: inputs that auto-generate typed outputs. For example, generate_uuid produces a variable with class: uuid automatically. Functions are pure generators — no configuration needed.

§3 Full Example

A complete test file with setup, typed variable outputs, and validation — services are inherited from the TCK definition file (§5).

kind: test name: available-notification version: "1.0" setup: - uses: mock_endpoint name: Expose mock endpoint for certificate pull with: id: "avail_cert_pull" method: GET path: "/api/v1/companycertificate/available" response_status: 200 response_body: businessPartnerNumber: ${{ env.provider_bpn }} certificateType: ${{ env.certificate_type }} steps: # Function step — no inputs, auto-generates a typed output - uses: generate_uuid name: Generate notification message ID outputs: avail_message_id: type: string class: uuid - uses: pull_data_filtered_from_precondition name: Discover notification endpoint with: service: provider_edc counter_party_id: ${{ env.provider_bpn }} counter_party_address: ${{ env.provider_address }} validate: - type: NOT_NULL output: edr_token - type: NOT_NULL output: dataplane_url outputs: edr_token: type: string class: auth_token dataplane_url: type: string class: dataplane_url - uses: http_call_dataplane name: Send available notification with: method: POST dataplane_url: ${{ vars.dataplane_url }} path: "/companycertificate/available" edr_token: ${{ vars.edr_token }} validate: - type: EQUALS output: status_code value: 200 - uses: wait_for_call name: Wait for SUT to pull certificate with: endpoint_id: "avail_cert_pull" timeout_s: ${{ env.sut_response_timeout }} validate: - type: EQUALS output: method value: "GET"

§4 Migration from v1 to v2

Complete guide for migrating existing TestLab YAML files to the new GitHub Actions-inspired syntax.

Quick Reference

v1 (current) v2 (new) type: http_call_dataplane → uses: http_call_dataplane description: "Send notification" → name: Send notification params: → with: @step_output → ${{ vars.step_output }} @config_value → ${{ env.config_value }} store_in_memory: { key: "value" } → outputs: { key: { class: type } } services: (top-level) → env: { services: [...] } validate: → validate: (unchanged)

Migration Rules

  1. Replace type: with uses: on every step
  2. Replace description: with name:
  3. Replace params: with with:
  4. Replace @variable_name with ${{ vars.variable_name }} when it comes from a previous step's outputs
  5. Replace @variable_name with ${{ env.variable_name }} when it comes from external configuration (connector addresses, BPNs, timeouts)
  6. Remove quotes from validation type values: "EQUALS" → EQUALS
  7. In services: replace type: with uses: and config: with with:
  8. Move services: from top-level into env: { services: [...] }
  9. Rename keyword: store_in_memory: → outputs: and add type annotations: outputs: { key: { class: <type> } } per ADR-0009
  10. Add class: to each env.variables entry to declare its semantic type

Steps — Before & After

v1 (current)
steps: - type: pull_data_filtered_from_precondition description: Discover notification endpoint params: service: provider_edc counter_party_id: "@provider_bpn" counter_party_address: "@provider_address" validate: - type: "NOT_NULL" output: edr_token store_in_memory: edr_token: "edr_token" dataplane_url: "dataplane_url"
v2 (new)
steps: - uses: pull_data_filtered_from_precondition name: Discover notification endpoint with: service: provider_edc counter_party_id: ${{ env.provider_bpn }} counter_party_address: ${{ env.provider_address }} validate: - type: NOT_NULL output: edr_token outputs: edr_token: type: string class: auth_token dataplane_url: type: string class: dataplane_url

Services — Before & After

v1 (current)

v2 (new)
env: services: - name: provider_edc uses: edc_connector_saturn with: base_url: ${{ env.provider_address }} dma_path: /management

§5 TCK Definition File

The TCK file defines the test suite context: environment variables (with typed classes), shared services, and preconditions. Individual test files inherit this context.

kind: tck name: certificate-management version: "1.0" description: > Validate CCMAPI certificate management per CX-0135 v3.1.0 author: Certificate Management Expert Group standards: - id: CX-0135 version: v3.1.0 tags: - CCM env: # Variables — available as ${{ env.variable_name }} variables: provider_address: class: url description: Provider EDC DSP endpoint required: true provider_bpn: class: bpn description: Provider BPNL required: true consumer_bpn: class: bpn description: Consumer BPNL (TestLab company) required: true certificate_type: class: string description: Certificate type to request default: "iso9001" sut_response_timeout: class: number description: Seconds to wait for SUT response default: 300 # Services — available by name in steps (service: provider_edc) services: - name: provider_edc uses: edc_connector_saturn with: base_url: ${{ env.provider_address }} dma_path: /management - name: testlab_edc uses: edc_connector_saturn with: base_url: ${{ env.testlab_management_url }} dma_path: /management # Preconditions — reusable policies/contracts preconditions: - id: ccmapi_usage_policy uses: policy name: CX-0135 ODRL usage policy with: dataspace_version: saturn policy_type: usage permissions: - action: use constraints: - left_operand: UsagePurpose operator: eq right_operand: "cx.ccm.base:1" # Test suite tests: - test: tests/request_certificate.yaml name: Request certificate with full header envelope - test: tests/available_notification.yaml name: Send available notification
